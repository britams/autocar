# 오토카 보고서

## 무엇을 했는지
- 컴퓨터 키보드로 오토카(자율주행 자동차 모형)를 조종하면서, 운동장 트랙을 주행하는 동안 오토카의 위치와 조도(빛의 밝기, CDS) 센서 값을 함께 기록하고 화면으로 확인할 수 있는 프로그램을 만듦.
1. 오토카의 바퀴 속도와 앞바퀴가 꺾인 각도(조향각)를 이용해서, 오토카가 지금 어디쯤 있는지를 계산으로 추정함. (자동차 위치 추정에 흔히 쓰이는 "자전거 모델"이라는 간단한 수학 모델을 사용함.)
2. 오토카에 달린 CDS(조도) 센서를 이용해서 빛의 밝기를 전압값(0~3.3V)으로 측정함.
3. 카메라 영상을 이용해서도 오토카의 이동 거리를 대략 계산하는 Visual Odometry(영상 기반 위치 추정) 기능을 추가로 만들어서, 위 1번의 오도메트리 계산 결과와 서로 비교·융합(fusion)해볼 수 있게 함.
4. 웹 브라우저에서 열 수 있는 조종 화면(대시보드)을 직접 만들어서, 키보드로 오토카를 조종하고 주행 경로와 센서 값을 실시간 그래프로 볼 수 있게 함.
5. "운전 시작"~"운전 정지" 버튼을 누르는 동안의 위치·속도·조향각·조도 값을 CSV 파일(엑셀로 열 수 있는 표 형식 파일)로 자동 저장하게 만듦.

---

## 프로젝트를 진행하면서 있었던 여러 고비 (문제 → 해결)

1. **오토카와 컴퓨터가 서로 연결이 안 됨**
   처음 사용하는 오토카라 Wi-Fi로 연결이 되지 않았음. 알아보니 이 오토카가 예전에 다른 컴퓨터의 인터넷망에 연결됐던 기록이 남아있어서, 지금 쓰는 컴퓨터와는 같은 네트워크가 아니었던 것이 원인이었음.
   → USB 케이블로 오토카(젯슨 보드, 오토카의 두뇌 역할을 하는 소형 컴퓨터)와 노트북을 직접 연결하는 방식으로 바꿔서 해결함.

2. **파일을 옮길 때(scp 명령어) 오류가 남**
   폴더 이름에 띄어쓰기가 있는 상태에서 여러 개의 CSV 파일을 한 번에 옮기려고 하니 "그런 파일이나 폴더가 없다"는 오류가 계속 발생함.
   → 파일 경로의 띄어쓰기 앞에 특수 기호(`\`)를 붙이고, 파일을 하나씩 옮기는 대신 폴더 전체를 통째로 복사하는 방식으로 바꿔서 해결함.

3. **회전한 각도가 실제보다 4배 크게 계산됨**
   오토카가 실제로는 90도만 돌았는데, 화면에는 360도 돈 것으로 표시됨.
   → 위치를 계산하는 수식에 넣는 값들(바퀴 축 사이의 거리, 최대로 꺾이는 조향각 등)이 실제 오토카의 치수와 정확히 맞지 않아서 생긴 오차였음. 실제로 돈 각도(90도)와 화면에 나온 각도(360도)를 비교해서 그 차이(4배)만큼 계산값을 나눠주는 보정값을 코드에 추가해서 해결함.

4. **키보드로 조종할 때 반응이 느림**
   → ① 방향키를 누르고 있는 동안 일정한 시간 간격으로만 신호를 보내던 방식을, 키를 누르거나 뗀 바로 그 순간에 즉시 신호를 보내는 방식으로 바꿈. ② 이전에 보낸 신호에 대한 응답이 오기도 전에 새 신호가 계속 쌓이는 문제가 있어서, 응답이 올 때까지는 새로 안 보내다가 응답이 오면 그 사이 눌린 가장 최근 키 상태를 바로 이어서 보내도록 고침. ③ 조도 센서 값을 읽어오는 작업과 오토카에 실제로 움직이라는 명령을 보내는 작업을 각각 별도의 스레드(thread)로 분리해서, 마치 별도의 MCU가 센서와 구동을 전담하는 것처럼 두 작업이 서로 기다리지 않고 동시에 처리되도록 만들어 반응 속도를 크게 개선함.

5. **조도 센서 값이 항상 1로 고정되어 있어서 제대로 된 값을 알 수 없었음**
   센서 연결 자체는 잘 되었다고 나오는데도 값이 전혀 바뀌지 않았음.
   → 센서가 값을 읽어올 수 있는 자리(채널)가 0번부터 7번까지 총 8개가 있는데, 8개를 하나씩 직접 확인해가며 손으로 센서를 가려보는 테스트를 해봄. 그 결과 실제로는 0번이 아니라 7번 자리에 연결되어 있다는 것을 알아내고 코드를 수정해서 해결함.

6. **조도 센서 값이 정수(딱 떨어지는 숫자)로만 나와서 미세한 변화를 알기 어려움**
   원래는 센서의 전압값을 밝기 단위(럭스)로 계산해서 반올림한 정수를 사용했는데, 그러다 보니 값이 몇 단계로만 뚝뚝 끊겨서 나왔음.
   → 밝기 단위로 계산하는 과정을 없애고, 센서가 직접 측정한 전압값(0.0~3.3V, 소수점 있는 값)을 그대로 사용하도록 바꿈.

7. **오토카에 달린 카메라가 인식되지 않음**
   Visual Odometry 기능은 만들어 두었지만, 실제 오토카에서 카메라가 인식되지 않아서 진짜 카메라 값이 아닌 가짜(테스트용) 값만 나왔음. 그래서 최종적으로 쓸 수 있는 위치 데이터는 오도메트리(바퀴 속도와 조향각 기반) 값뿐이었음. (실제로 카메라까지 함께 썼다면 더 정확한 위치를 얻을 수 있었을 것이라고 생각함.)
   → 카메라가 실제로 사진 한 장을 읽어오는지 확인한 뒤에만 "연결 성공"이라고 판단하도록 코드를 고쳐서, 최소한 카메라가 진짜 연결되었는지 아닌지는 정확하게 알 수 있도록 개선함.

8. **수동 조종과 자동 주행 모드를 바꿀 때마다 CSV 파일이 여러 개로 쪼개짐**
   키보드 조종과 자동 주행 모드를 서로 바꿀 때마다 그 순간 CSV 파일이 끝나고 새 파일이 또 생겨서, 원래는 한 번의 주행 기록인데 여러 개의 파일로 나뉘어 저장되는 문제가 있었음.
   → 모드를 바꿀 때는 오토카만 잠깐 멈추게 하고 CSV 파일은 계속 열어둔 채로 두었으며, 실제로 "운전 정지"나 "자동 주행 정지" 버튼을 눌렀을 때만 파일이 저장되고 닫히도록 고쳐서, 모드를 바꿔도 하나의 파일에 계속 이어서 기록되게 함.

9. **배터리를 쓸 때 전압값과 전류값이 떨어지는 현상이 나타남**
   → 오토카를 다른 개체로 교체하는 작업을 진행함.

10. **젯슨 보드에 저장된 CSV 파일이 계속 쌓이고 지워지지 않음**
    프로그램을 다시 실행해도 이전에 모은 CSV 파일들이 오토카(젯슨 보드) 안에 계속 남아 있어서, 직접 하나씩 지우지 않으면 파일이 계속 쌓이는 문제가 있었음.
    → 프로그램을 실행할 때마다 이전에 저장돼 있던 CSV 파일을 자동으로 모두 지우고 새로 시작하도록 코드를 추가해서 해결함.

## 개선하면 좋을 점 (향후 개선사항)
- 카메라를 실제로 연결해서, Visual Odometry로 계산한 위치와 오도메트리(바퀴 속도 기반) 위치를 서로 비교·융합(fusion)해서 사용한다면 위치 추정이 더 정확해질 것이라고 생각함.
- 지금은 "운전 정지" 버튼을 눌러야만 CSV 파일이 저장되는데, 일정한 시간이나 거리마다 자동으로 파일을 나눠서 저장하는 기능이 있으면 오래 주행해도 데이터를 잃어버릴 위험이 줄어들 것 같음.
- 대시보드 화면에서 CSV 파일을 오토카로부터 바로 내려받을 수 있는 버튼을 만들면, 매번 파일 옮기는 명령어를 직접 입력하지 않아도 되어서 더 편리할 것 같음.
- 카메라를 이용하여 벽을 따라 주행하는 자율주행 기능(벽 추종, wall-following)을 추가로 구현한다면 프로젝트의 활용도를 더욱 높일 수 있을 것으로 생각됨.

## 아쉬운 점
- 오토카의 하드웨어(카메라, 조도 센서가 연결된 자리, Wi-Fi 연결 상태 등)에 대한 정보를 미리 충분히 알지 못해서, 하나씩 직접 확인해보며 원인을 찾는 데 생각보다 시간이 많이 걸렸음.
- Visual Odometry 관련 코드는 완성해두었지만, 실제 카메라 하드웨어 문제로 이 기능이 실제로 얼마나 정확한지 검증해보지 못한 점이 아쉬움.
- 오토카의 반응 속도를 더 빠르게 만들 수 있는 여러 방법(코드 경량화, MCU를 추가로 도입해서 SBC(오토카의 메인 연산 장치)에 걸리는 계산 부담을 나누는 방법, 더 좋은 배터리 사용 등)을 충분히 시도해보지 못한 점이 아쉬움.
- 오토카를 무선(예: 휴대폰의 핫스팟 기능을 이용해 임시 공유기처럼 활용하는 방식)으로 조종하는 방안을 시도해보고자 하였으나, 시간상의 제약으로 오토카의 무선 IP 주소를 확인하는 작업까지 진행하지 못한 점이 아쉬움으로 남음.

## 느낀 점
- 실제 하드웨어를 다루면서, 코드를 잘 짜는 것 못지않게 "실제로 무엇이 어떻게 연결되어 있고 동작하는지"를 직접 확인하고 테스트해보는 과정이 훨씬 더 중요하다는 것을 느낌.
- 네트워크 연결, 센서 자리, 반응 속도 등 예상하지 못한 문제가 계속 나왔지만, 원인을 하나씩 좁혀가면서 해결하는 과정에서 문제를 스스로 찾아내고 고치는 능력을 많이 기를 수 있었음.
- 오토카가 어떠한 방식으로 동작하고 데이터를 처리하는지 이해하게 되었으며, 이러한 문제 해결 접근 방식을 향후 다른 제어 관련 과제에도 동일하게 적용할 수 있을 것이라는 확신을 얻음.
- 오토카 제어에 필요한 복잡한 코드를 AI 에이전트를 활용하여 보다 신속하고 정확하게 구현할 수 있음을 체감하였으며, 이러한 방식이 오토카뿐만 아니라 다른 장치를 다루는 프로젝트에도 확장 적용될 수 있을 것으로 생각됨.
