# 7/3 과제 — 오토카 운동장 트랙 돌기

## 목표
운동장 트랙을 돌면서 CDS(조도) 센서 값과 오토카의 위치 정보를 함께 수집하고,
웹 브라우저 대시보드에서 실시간으로 시각화하며, CSV 파일로 저장한다.

---

## 현재 구현 상태 (이 문서만 보고 바로 이어서 작업할 수 있도록 정리함)

### 1. 파일 구성
프로젝트 폴더: `/home/eohyun_ee/autocar/오토카 운동장 돌기/`

| 파일/폴더 | 역할 |
|---|---|
| `track_dashboard.py` | 메인 실행 파일. Flask 웹 서버 + 대시보드 화면 + 하드웨어 제어. |
| `track_odometry.py` | 모터 속도 + 조향각 기반 위치 추정(오도메트리, 자전거 모델) 모듈. |
| `track_vision.py` | 카메라 영상 기반 위치 추정(Visual Odometry, 옵티컬 플로우) 모듈. 라인트레이싱 용도 아님, 위치 추정 보완용. |

### 2. 실행 환경 / 네트워크
- 오토카는 **젯슨(Jetson) 보드 + USB 케이블로 노트북과 직접 연결**해서 사용한다 (Wi-Fi 아님).
  - USB 연결 시 젯슨 보드 쪽 IP는 보통 `192.168.55.1`, 노트북 쪽은 `192.168.55.100`으로 잡힌다 (Jetson USB 디바이스 모드 기본값).
  - 예전에 쓰던 주소 `192.168.0.57`(같은 공유기 Wi-Fi 연결 기준)은 현재 USB 직결 환경에서는 사용하지 않는다.
- 오토카 로그인: `soda@192.168.55.1`
- 코드를 오토카로 옮길 때 (WSL 터미널):
  ```
  scp -r "/home/eohyun_ee/autocar/오토카 운동장 돌기" soda@192.168.55.1:~/
  ```
- 오토카 접속 및 실행 (WSL 터미널 → 오토카 터미널):
  ```
  ssh soda@192.168.55.1
  cd ~/오토카\ 운동장\ 돌기
  python3 track_dashboard.py
  ```
- 브라우저 접속: `http://192.168.55.1:5000`
- 오토카 "파워선"(모터/CAN 전원)이 빠져 있고 USB로 젯슨 보드만 연결된 상태에서도 실행되어야 하며, 실제로 각 하드웨어(오토카 구동부, CDS 센서, 카메라)가 연결되지 않으면 해당 부분만 자동으로 "시뮬레이션 모드"로 대체되고 프로그램 전체는 정상 동작한다.

### 3. 하드웨어 연동
- 오토카 구동: `pop.Pilot.AutoCar` 클래스 사용 (`car.steering`(속성, -1~1), `car.setSpeed()`(0~99), `car.forward()`/`car.backward()`/`car.stop()`).
- CDS(조도) 센서: `pop.Cds` 클래스, **SPI ADC 채널 7번**에 연결되어 있음 (`CDS_ADC_CHANNEL = 7`, 실측으로 확인함 — 0~7 채널을 전수 테스트해서 찾음).
  - 값은 `cds_sensor.readVoltAverage()`로 읽은 **아날로그 전압값(0.0~3.3V)**을 그대로 사용한다. (`readAverage()`가 반환하는 럭스 환산 정수값은 쓰지 않음 — 값이 계단식으로만 나오는 문제가 있었음.)
- 카메라: `pop.Util.gstrmer()` GStreamer 파이프라인 우선 시도, 실패 시 일반 웹캠(`cv2.VideoCapture(0)`) 시도. 실제 프레임이 한 장이라도 읽혀야 "연결 성공"으로 판정한다 (단순 `isOpened()`만으로는 오탐지 발생했었음).

### 4. 위치 추정 (오도메트리) 보정값
`track_odometry.py`의 `TrackOdometry` 클래스 상수:
- `WHEELBASE_M = 0.20` (앞바퀴축-뒷바퀴축 거리, 추정값 — 실측 안 됨, 개선 필요)
- `MAX_SPEED_MPS = 1.0` (speed=99일 때 실제 속도, 추정값 — 실측 안 됨, 개선 필요)
- `MAX_STEER_ANGLE_DEG = 20.0` (steering=±1일 때 실제 조향각, 추정값)
- `STEER_SIGN = 1` (조향 부호 방향)
- `TURN_CORRECTION_FACTOR = 4.0` — **실측으로 확정된 값**. 실제 90도 회전 시 웹뷰에 360도로 표시되는 것을 확인하고, 그 비율(4배)만큼 회전 속도 계산값을 나누도록 추가한 보정 계수.
  - 재보정 공식: `TURN_CORRECTION_FACTOR = (웹뷰 표시 회전각) / (실제 회전각)` 비율만큼 기존 값에 곱해서 조정.

카메라 기반 `track_vision.py`의 `VisualOdometry`는 `METERS_PER_PIXEL = 0.002`(추정값, 미보정) 등을 사용하며, 현재 오토카 카메라가 인식되지 않아 시뮬레이션 값만 나오는 상태 — 실제 카메라 연결 시 재보정 필요.

### 5. 조종 방식
- **키보드 방향키** (조이스틱 아님): ↑ 전진, ↓ 후진, ← 좌회전, → 우회전. 키를 누르는 즉시 서버로 전송되며, 뗄 때도 즉시 정지 신호 전송.
- **자동 주행**: 슬라이더로 지정한 속력+조향값으로 일정하게 도는 단순 이동 프로그램 (원형 트랙 주행용).
- 수동/자동은 동시에 켤 수 없고, 서로 전환 시 오토카만 잠깐 멈추고 **CSV 기록은 끊기지 않는다** (내부적으로 `/stop`으로 전환, `/drive/stop`·`/auto/stop`은 아님).
- "운전 시작"(수동) 또는 "자동 주행 시작" 버튼 = 조종 활성화 + CSV 기록 시작.
- "운전 정지"/"자동 주행 정지" 버튼 = 정지 + CSV 기록 종료(파일 저장).
- 워치독: `/control` 신호가 `WATCHDOG_TIMEOUT_SEC(0.3초)` 이상 안 오면 자동 정지.

### 6. 반응속도 최적화 (스레드 구조)
GIL(파이썬은 한 순간에 한 스레드만 실행) 경합을 최소화하기 위해, 아래 항목들을 모두 **전용 배경 스레드**로 분리했다. `/control` 같은 웹 요청 처리 스레드는 각 항목의 "목표 상태"만 빠르게 갱신하고, 실제 하드웨어 통신은 해당 전용 스레드가 전담한다.
1. `_watchdog_loop` — 조종 워치독 (0.05초 주기)
2. `_drive_writer_loop` — 목표 상태(방향/속력/조향)를 오토카에 실제로 전달 (`DRIVE_WRITE_INTERVAL_SEC = 0.02초` 주기, 초당 최대 50회)
3. `_cds_reader_loop` — CDS 센서 실측값을 캐시에 갱신 (`CDS_READ_INTERVAL_SEC = 0.1초` 주기, `CDS_SAMPLE_COUNT = 8` 샘플)
4. `_sampling_loop` — 위치·CDS 값을 합쳐 화면용 trail에 기록 + 녹화 중이면 CSV에 한 줄 기록 (`SAMPLE_INTERVAL_SEC = 0.1초` 주기)
5. `_camera_loop` — 카메라 프레임 캡처 + Visual Odometry 갱신 (별도 스레드, 웹 서버가 뜬 뒤에 카메라를 엶)
6. 키보드 입력 자체도 프론트엔드에서 "누르는/떼는 순간 즉시 전송 + 이전 요청이 끝나기 전엔 새로 안 보내고 최신 상태만 이어보내는" 방식으로 요청 밀림을 방지함.

### 7. Flask 라우트 목록
| 경로 | 설명 |
|---|---|
| `GET /` | 대시보드 HTML 페이지 |
| `GET /status` | 하드웨어 연결 상태 (`has_car`, `has_cds`, `has_cam`) |
| `POST /control` | 방향키 상태(x=조향, y=속력방향) 적용. 자동주행 중엔 무시됨 |
| `POST /stop` | 오토카만 즉시 정지 (CSV는 안 건드림) — 모드 전환/긴급정지용 |
| `POST /ping` | 워치독 갱신 |
| `POST /drive/start` | 수동 조종 활성화 + CSV 기록 시작(이미 기록 중이면 기존 파일 유지) |
| `POST /drive/stop` | 정지 + CSV 기록 종료 |
| `POST /auto/start` | 자동 주행 시작(`speed`, `steering`) + CSV 기록 시작 |
| `POST /auto/stop` | 자동 주행 정지 + CSV 기록 종료 |
| `GET /track` | 최근 위치/CDS trail(최대 400개) + Visual Odometry 위치 반환 (대시보드가 0.3초 간격으로 폴링) |

### 8. CSV 저장 정책
- 저장 위치: `csv 파일 모음/track_data_YYYYMMDD_HHMMSS.csv`
- 컬럼: `timestamp_unix, elapsed_sec, x_m, y_m, heading_deg, vo_x_m, vo_y_m, speed, steering, cds_value`
  - `x_m, y_m, heading_deg`: 모터 기반 오도메트리 위치/방향
  - `vo_x_m, vo_y_m`: 카메라 기반 Visual Odometry 위치 (참고용, 카메라 미인식 시 0)
  - `cds_value`: CDS 아날로그 전압값 (V)
- **대시보드를 새로 실행할 때마다 `csv 파일 모음` 폴더 안의 기존 `.csv` 파일을 전부 삭제**하고 새로 시작한다 (`CLEAR_CSV_ON_STARTUP = True`). 전원을 안 껐어도 스크립트를 재실행하면 지워지므로, 재실행 전엔 필요한 CSV를 노트북으로 미리 옮겨둘 것.
  - CSV 가져오기 (WSL 터미널):
    ```
    scp -r "soda@192.168.55.1:~/오토카\ 운동장\ 돌기/csv\ 파일\ 모음" "/home/eohyun_ee/autocar/오토카 운동장 돌기/"
    ```

### 9. 알려진 한계 / 향후 개선 필요 사항
- 카메라가 실제로 인식되지 않아 Visual Odometry는 코드만 구현되어 있고 실측 검증이 안 된 상태.
- `WHEELBASE_M`, `MAX_SPEED_MPS`, `MAX_STEER_ANGLE_DEG`, `METERS_PER_PIXEL` 등은 실측이 아닌 추정값이라 정밀도가 떨어짐 (자세한 내용은 `csv 파일 모음/최종정리보고서.md` 참고).
- 코드 스타일 요구사항: 파일명·변수명·함수명은 항상 프로젝트에 어울리게 짓고, 불필요한 파일은 만들지 않으며 생기면 삭제할 것. 모든 코드에는 원리를 설명하는 한글 주석(특히 숫자 상수를 바꿨을 때 동작이 어떻게 달라지는지)을 충실히 달 것 — 작성자가 코딩 초보자이므로 쉬운 설명 필수.
