# 오토카 보고서

## 무엇을 했는지
- 컴퓨터 키보드를 이용하여 오토카를 조종하고, 운동장 트랙을 주행하면서 위치 및 조도(CDS) 센서 데이터를 수집·시각화하는 프로젝트를 수행함.
1. 오토카의 모터 속도와 앞바퀴 조향각을 이용하여(자전거 모델, bicycle model) 오토카의 위치를 추정함.
2. 오토카의 CDS(조도) 센서를 통해 조도값을 아날로그 전압(0~3.3V)으로 측정함.
3. 카메라 영상을 이용한 Visual Odometry(카메라 기반 위치 추정) 기능을 함께 구현하여, 모터 기반 위치 추정을 보완할 수 있도록 설계함.
4. 웹 브라우저 기반 대시보드(Flask)를 제작하여 키보드로 오토카를 조종하고, 실시간으로 주행 경로 및 센서 값을 화면에서 확인할 수 있도록 구현함.
5. "운전 시작"~"운전 정지" 버튼을 통해 위치·속도·조향각·CDS 값을 CSV 파일로 자동 저장하도록 구현함.

---

## 프로젝트를 진행하면서 있었던 여러 고비 (문제 → 해결)

1. **오토카와 컴퓨터 간 연결 불가**
   신규로 사용하는 오토카였기에 Wi-Fi 연결이 되지 않았음. 확인 결과, 해당 오토카가 이전에 다른 컴퓨터의 네트워크에 접속했던 이력이 남아 있어, 현재 사용 중인 컴퓨터와 동일한 네트워크 환경이 아니었던 것이 원인으로 확인됨.
   → USB 케이블을 통해 오토카(젯슨 보드)와 노트북을 직접 연결하는 방식으로 전환하여 해결함.

2. **파일 전송(scp) 오류**
   폴더명에 공백이 포함된 상태에서 다수의 CSV 파일을 한 번에 전송하기 위해 와일드카드(`*.csv`)를 사용하였으나, "No such file or directory" 오류가 반복적으로 발생함.
   → 원격 경로의 공백 앞에 백슬래시(`\`)를 추가하고, 와일드카드 대신 폴더 전체를 일괄 복사하는 방식(`scp -r`)으로 변경하여 해결함.

3. **회전각 계산값이 실제값보다 4배 크게 산출됨**
   오토카가 실제로는 90도 회전하였으나, 웹뷰 대시보드에는 360도 회전한 것으로 표시됨.
   → 위치 추정에 적용한 자전거 모델 계산 과정에서, 축간거리 및 최대 조향각 등의 추정값이 실제 오토카의 제원과 상이하여 발생한 오차로 확인됨. 실측값(90도) 대비 표시값(360도)의 비율(4배)만큼 회전 속도값을 나누어주는 보정 계수(`TURN_CORRECTION_FACTOR`)를 위치 추정 로직에 추가하여 해결함.

4. **키보드 조종 시 반응속도 지연**
   → ① 방향키 입력 유지 중 일정 주기로만 서버에 신호를 전송하던 방식을, 입력 및 해제 시점에 즉시 전송하는 방식으로 변경함. ② 이전 요청이 완료되기 전 새로운 요청이 누적되는 문제를 방지하기 위해, 요청을 순차적으로 하나씩만 처리하되 처리 완료 시점의 최신 입력 상태를 즉시 반영하는 방식으로 개선함. ③ CDS 센서 값 읽기와 오토카 구동 명령 전달을 각각 전용 스레드로 분리하여, 별도의 MCU가 센서 및 구동을 전담하는 구조를 모사함으로써 조종 처리와 완전히 독립시켜 반응속도를 대폭 향상시킴.

5. **CDS 값이 1로 고정되어 유의미한 값 확인 불가**
   센서 연결 자체는 정상적으로 이루어졌다는 응답이 있었으나, 값이 전혀 변화하지 않는 현상이 발생함.
   → 8개 채널(0~7)을 전수 점검하며 센서를 수동으로 가리는 테스트를 진행한 결과, 실제 연결 채널이 0번이 아닌 7번임을 확인하여 채널 번호를 수정함으로써 해결함.

6. **CDS 값이 정수 단위로 출력되어 세밀한 변화 확인 곤란**
   기존에는 전압값을 럭스(lux, 조도 단위)로 환산 후 반올림한 정수값을 사용하여, 값이 불연속적인 단계로만 표시되는 문제가 있었음.
   → 럭스 환산 과정을 생략하고, 센서가 실측한 아날로그 전압값(0.0~3.3V, 소수점 단위)을 그대로 사용하도록 변경함.

7. **오토카 카메라 미인식**
   Visual Odometry 기능은 구현을 완료하였으나, 실제 오토카에서 카메라가 인식되지 않아 시뮬레이션 모드로만 동작하였고 실측 카메라 데이터는 확보하지 못함. 이에 따라 최종적으로는 모터 조향각 및 속도 기반의 위치 추정값만 유효한 데이터로 활용됨.
   → 카메라로부터 실제 프레임 수신 여부를 확인한 이후에만 "연결 성공"으로 판정하도록 로직을 수정하여, 최소한 연결 실패 여부를 정확히 감지할 수 있도록 개선함.

8. **수동/자동 조종 모드 전환 시 CSV 파일 분할 저장**
   키보드 조종과 자동 주행 모드를 상호 전환할 때마다 CSV 파일이 종료되고 새로 생성되어, 하나의 주행 기록이 여러 파일로 분산 저장되는 문제가 있었음.
   → 모드 전환 시에는 오토카 구동만 일시 정지하고 CSV 파일은 종료하지 않도록 수정하였으며, "운전 정지"/"자동 주행 정지" 버튼을 실행한 경우에만 파일이 종료되도록 개선하여, 모드 전환 여부와 무관하게 하나의 파일에 연속으로 기록되도록 함.

9. **배터리 사용 중 전압·전류값 저하 현상**
   → 오토카 교체 작업을 진행함.

## 개선하면 좋을 점 (향후 개선사항)
- 카메라를 실제로 연결하여 Visual Odometry 결과를 모터 기반 위치 추정값과 상호 비교·융합(fusion)한다면, 위치 추정의 정확도를 한층 높일 수 있을 것으로 판단됨.
- 축간거리, 최대 조향각, 최대 속도 등의 제원을 줄자 및 스톱워치로 직접 실측하여 반영한다면, 현재의 결과 비율 기반 보정 방식보다 근본적으로 정밀한 위치 추정이 가능할 것으로 예상됨.
- CDS 전압값과 실제 조도(lux) 단위 간의 상관관계를 조도계를 이용해 실측·비교하여 캘리브레이션한다면, 전압값만으로는 파악하기 어려운 실제 밝기 정보까지 분석에 반영할 수 있을 것으로 보임.
- 현재는 운전 정지 버튼 조작 시에만 CSV가 저장되므로, 일정 시간 또는 거리 단위로 자동 분할 저장하는 기능을 추가한다면 장시간 주행 시에도 데이터 유실 위험을 줄일 수 있을 것으로 사료됨.
- 대시보드 내에 CSV 파일을 오토카로부터 직접 내려받을 수 있는 기능을 추가한다면, 매 회 별도의 scp 명령 입력 없이도 파일을 확보할 수 있어 편의성이 향상될 것으로 예상됨.
- 위치정보가 정확하지 않음. 보정값을 더욱 주고 카메라를 이용해서 오도메트리 측정을 추후에 하길 바람. 

## 아쉬운 점
- 오토카 하드웨어(카메라, CDS 센서 채널, Wi-Fi 연결 상태 등)에 대한 사전 정보가 충분하지 않아, 각 요소를 순차적으로 점검하며 원인을 파악하는 데 예상보다 많은 시간이 소요됨.
- Visual Odometry 관련 코드는 구현을 완료하였으나, 하드웨어 문제로 인해 실제 성능 및 정확도를 검증하지 못한 부분이 아쉬움으로 남음.
- 축간거리·조향각 등 물리적 제원을 직접 실측하지 못하고 결과 비율에 기반해 보정한 부분이 있어, 위치 추정의 정밀도 측면에서 완전하다고 보기는 어려움.
- 오토카의 반응속도를 한층 더 향상시키기 위한 다양한 방안(코드 경량화, MCU 도입을 통한 SBC 연산 부담 경감, 고성능 배터리 적용 등)을 충분히 검토하지 못한 점이 아쉬움으로 남음.

## 느낀 점
- 실제 하드웨어를 다루는 과정에서, 코드 작성 능력 못지않게 실제 연결 상태와 동작 방식을 직접 확인하고 검증하는 과정의 중요성을 체감함.
- 네트워크 연결, 센서 채널, 반응속도 등 예상하지 못한 문제가 지속적으로 발생하였으나, 원인을 단계적으로 좁혀가며 해결하는 과정을 통해 실질적인 디버깅 역량을 함양할 수 있었음.
- 조향각·축간거리와 같이 실측이 요구되는 값들은, 우선 잠정값을 적용한 뒤 실제 테스트 결과와 비교하며 점진적으로 보정해나가는 방식이 실무적으로 유효함을 습득함.
