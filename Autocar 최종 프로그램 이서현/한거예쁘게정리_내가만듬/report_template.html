TensorRT 가속

Jetson Nano에서 YOLOv3-tiny를 TensorRT(NVIDIA 모델 최적화 도구)로 변환해서 5배 빠른 실시간 탐지를 만들었다. 6 FPS는 영상이 뚝뚝 끊겨 실시간 대응이 어려운 수준, 30 FPS는 사람 눈에 부드럽게 느껴지는 "실시간" 수준이다.

왜 YOLOv4가 아니라 YOLOv3-tiny인가

YOLOv4(약 6,400만 파라미터)는 YOLOv3-tiny(약 880만 파라미터)보다 정확하지만 7배 무겁다. Jetson Nano에 YOLOv4를 올리면 FPS가 한 자릿수로 떨어질 것으로 예상되는데, 자율주행차가 초당 2~3프레임으로 판단하면 장애물을 발견했을 때 이미 늦다. 정확도를 다소 포기하더라도 실시간이 가능한 쪽을 택했다.

실기 디버깅

  • 차가 안 움직이던 원인: setSpeed()가 "이미 주행 중"일 때만 반영되는 함수라 정지 상태에서 무효였음 → forward(speed)로 직접 전달하도록 수정
  • 인식 떨림으로 정지↔전진이 반복되던 문제: 스무딩(급격한 변화 방지) + 디바운스(짧은 오차 무시)로 해결

사람 놓치면 카메라 탐색

사람을 놓치면 카메라 서보를 90°(중앙)→0°→180°→90° 순서로 천천히 왕복(최대 20회)하며 찾는다. 인식이 깜빡거릴 때마다 탐색이 리셋되지 않도록, 연속 3프레임 이상 확인돼야 "진짜로 찾음"으로 인정한다.

🛣️ 라인트레이싱 (OpenCV)

카메라 화면에서 흰 선을 찾아 화면 중앙에 오도록 조향한다. "주변 트랙보다 상대적으로 밝은가"를 기준으로 판단(적응형 임계값)해서 날씨·노출이 바뀌어도 안정적으로 동작하게 만들었다.

오인식 방지

  • 인식 범위(ROI)를 화면 아래쪽 50~55%로 좁혀 하늘·건물을 분석 대상에서 제외
  • 일정 크기 이상 뭉친 덩어리만 선으로 인정해 트랙 균열 노이즈 제거
  • 끊어진 점선 조각들을 먼저 이어붙인 뒤 하나의 선으로 판단
  • "양옆 두 선 사이 중앙 유지" 대신, 실제 트랙 구조(중앙 점선 하나)에 맞춰 가장 크게 잡힌 흰 선 덩어리 하나를 추적하는 방식으로 확정
  • LINE 모드 진입 시 카메라를 좌우 정중앙(pan 90°) + 아래쪽(tilt -10°)으로 자동 정렬

디버그 화면 — 스트리밍 대신 새로고침

인식 중인 화면(선을 초록색으로 표시)을 조종 화면에 내장했다. 계속 스트리밍하면 처리 부하가 커져 반응 속도가 크게 느려지기 때문에, 새로고침 버튼을 누를 때만 사진 한 장을 받아오는 방식을 택했다.

장애물 대응 — 정지 후 재출발

라인트레이싱 중 장애물을 만나면 일단 멈췄다가 사라지면 다시 출발한다. 후진·회전 회피는 하지 않는데, 자세한 이유는 02장에서 다룬다.

🚦 라이다 · AI 역할 분담 — 안전 우선순위 교통정리

라이다(장애물 감지)와 AI 인식(YOLO/OpenCV)은 같은 층위에서 경쟁하지 않는다. 라이다가 항상 최우선으로 개입하는 위계로 나눴다.

최종 결정권자
라이다

매 프레임 가장 먼저 좌/우 근접 장애물을 확인한다. 감지되면 모드와 상관없이 그 프레임에서는 AI 인식 로직을 아예 실행하지 않고 회피(또는 정지) 동작만 한다.

항상
우선
평상시 주행 담당
YOLO / OpenCV

장애물이 없을 때만 각 모드(사람 추적 / 라인 추적)의 인식 로직이 실행된다. 딱 하나 예외는 "MANUAL_NOLIDAR" — 벽에 붙여 조종해야 할 때만 라이다 개입을 끌 수 있다.

라이다가 "멈출지 말지"를 결정하는 안전 담당, AI는 그 위에 얹히는 평상시 주행 담당.

이렇게 나누지 않고 두 판단을 동시에 반영했다면, "AI는 전진하라는데 라이다는 멈추라"는 상황에서 어느 쪽을 따를지 애매해졌을 것이다.

⚙️ 스레드 / 프로세스 구조

"동시에 여러 일을 처리하는 방식"은 세 층으로 나뉜다.

메인 스레드
카메라 읽기 · YOLO 인식 · 라인트레이싱 인식 · 모터 제어. YOLO의 GPU 연산(pycuda)이 "처음 초기화된 스레드에서만 동작"하는 제약이 있어 반드시 여기서 돌려야 했다.
별도 스레드 1개
폰과 통신하는 Flask 웹서버. 메인 스레드가 카메라/모터로 바쁜 동안에도 폰의 모드 전환·조향 명령을 받아야 해서 분리했다.
별도 프로세스 1개
라이다는 스레드가 아니라 완전히 다른 프로그램(프로세스)으로 분리했다 — 이유는 02장 참고. 스레드로 돌리면 GIL 문제로 프로그램 전체가 죽었다.

가벼운 통신(폰)은 스레드로, 무거운 하드웨어 작업(카메라/YOLO)은 메인 스레드로, 불안정한 하드웨어 작업(라이다)은 아예 별도 프로세스로 — 나눈 기준은 "안정성이 얼마나 중요한가"였다.

YOLO 가속 여정 — TensorRT까지 가는 길

YOLOv4-tiny로 먼저 시도했다가 이 Jetson의 구버전 OpenCV(4.3.0)가 YOLOv4의 최신 구조를 지원하지 않아 실패했고, YOLOv3-tiny + CUDA DNN도 빌드 문제로 막혀 결국 TensorRT 변환으로 우회했다. 그 과정에서 만난 7단계 문제:

#문제원인해결
1onnx==1.4.1 설치 실패onnx·protobuf 버전 불일치onnx==1.9.0 + 최신 protobuf로 재설치
2nvcc not foundCUDA 툴킷 경로가 PATH에 없음~/.bashrc에 CUDA 경로 추가
3pycuda 설치 스크립트 중단스크립트 내부 로직 버그pip3 install pycuda로 직접 설치
4변환 스크립트 "file not found"파일명에 해상도(-416) 누락yolov3-tiny-416.cfg/.weights로 이름 통일
5libyolo_layer.so 없음plugins 폴더 빌드(make) 안 함cd plugins && make
6make 자체가 실패Makefile 정규식의 #을 주석으로 오인grep -Po 대신 awk로 대체
7build_serialized_network 없음설치된 TensorRT(7.1.3)가 그 함수를 지원하는 8버전보다 낮음TensorRT 7 API인 build_engine으로 수정
결과 — yolov3-tiny-416.trt 엔진 생성 성공, 실시간 카메라 테스트에서 FPS 30.1 확인 (기존 6.0 대비 5배).

흰 선을 아예 못 찾음

해결
문제

"밝기 170 이상 = 흰색"이라는 절대 기준이, 흐린 날 어둡게 찍힌 화면에서는 흰 선도 못 넘어 인식 자체가 안 됨

해결

"주변 트랙보다 상대적으로 밝은가"를 보는 적응형 임계값(adaptiveThreshold)으로 교체. 날씨·노출이 바뀌어도 안정적

하늘/건물/트랙 균열 오인식

해결
문제

인식 범위가 화면 위쪽 25%부터라 하늘·건물 윤곽선까지 포함, 트랙 균열도 밝기 차이로 노이즈로 걸림

해결

인식 범위를 화면 아래 55%(트랙 바닥만)로 축소 + 일정 크기 이상 뭉친 덩어리만 선으로 인정

점선이 끊어져 조각이 너무 작음

해결
문제

트랙 페인트가 닳아 점선처럼 끊어져 있어, 조각 하나하나가 크기 필터에 걸려 다 무시됨

해결

끊어진 조각들을 먼저 이어붙인(dilate) 다음 필터링하도록 수정

카메라 프레임이 밀림

해결
문제

GStreamer appsink에 "오래된 프레임 버리기" 설정이 없어, 처리가 느려지면 몇 초 전 화면이 계속 나옴

해결

appsink drop=true max-buffers=1 sync=false 추가. 카메라를 여는 코드가 한 곳뿐이라 다른 모드(YOLO)에도 동일 적용됨

안정성 문제 — 라이다와 씨름한 기록

라이다를 스레드로 돌리면 프로그램 전체가 죽음

해결
문제

파이썬 GIL(한 번에 딱 한 스레드만 코드 실행) 특성상, 라이다의 네이티브 코드가 하드웨어 응답을 기다리며 GIL을 계속 붙잡고 있으면 카메라/YOLO가 도는 메인 스레드까지 통째로 멈춤

해결

라이다를 스레드가 아니라 완전히 별도의 프로세스로 분리. 프로세스는 각자 자기만의 GIL을 가져 서로 영향받지 않음

카메라+라이다 동시 자원 사용 시 멈춤

해결
문제

라이다 연결 순간 카메라/GPU가 이미 자원을 쓰고 있으면 연결 과정에서 멈추는 경우 발생

해결

연결 전 대기 시간 + 연결 시도 자체에 제한시간을 둬서, 시간 안에 안 되면 라이다 없이도 나머지는 정상 작동

별도 프로세스로 분리해도 그 자체가 멈춤

해결
문제

라이다를 완전히 분리한 뒤에도, 그 프로세스가 connect() 도중 응답 없이 완전히 멈춰버리는 일이 실제로 있었음

해결

감시자(부모)–작업자(자식) 이중 구조. 자식이 10초 안에 연결 못 하면 부모가 강제로 죽이고 새 자식으로 재시도

사람 여러 명 있으면 다른 사람으로 갈아탐

해결
문제

get_person_box()가 매 프레임 "제일 큰(가까운) 사람"만 골라서, 더 가까운 사람이 나타나면 추적 대상이 바뀜

해결

직전에 추적하던 사람 위치와 가장 가까운 사람을 우선 채택하도록 변경. 처음 찾을 때만 예외적으로 가장 큰 사람 선택

아직 못 푼 것 — 로고 오인식과 위치 복귀

트랙 위 로고를 선으로 착각

미해결
문제

"가장 크게 잡힌 흰 덩어리 = 선"이라는 규칙이라, 원형 로고처럼 크고 흰 무늬가 실제 선보다 크게 잡히면 그쪽으로 잘못 인식

시도했지만…

"크기 무시하고 직전 위치와 가장 가까운 것"으로 바꿔봤지만, 진짜 선을 눈앞에 두고도 못 찾는 더 나쁜 문제가 생겨 기존 방식(크기 우선 + 위치 참고)으로 되돌림

장애물 피한 뒤 정확히 원래 선으로 복귀

미해결
문제

GPS도 바퀴 오도메트리도 없어서 차가 "지금 정확히 어디에 있는지" 알 방법이 없음. 화면 속 좌표 기억은 실제 위치 정보가 아님

대체 방안

정밀 복귀는 포기하고, 장애물 감지 시 정지 → 사라지면 재출발로 설계 자체를 단순화. 회피 동작(후진/회전)도 하지 않기로 함 — 정확한 복귀가 안 되는 상황에서 회피까지 하면 트랙을 더 벗어날 위험만 커지기 때문

공통 원인 — 두 미해결 문제 모두 "화면에 지금 보이는 것"에만 의존하는 인식의 한계에서 온다. 03장, 04장에서 이어서 정리한다.
03

결과 및 결론

기능 1
사람 따라가기

YOLOv3-tiny + TensorRT (딥러닝) · 라이다 감지 시 회피 동작

기능 2
라인트레이싱

OpenCV 영상처리 (규칙 기반) · 라이다 감지 시 정지 후 재출발

두 가지 큰 트레이드오프

  • 정확도 vs 속도 — 더 정확한 YOLOv4 대신, Jetson Nano에서도 실시간(30 FPS)이 가능한 YOLOv3-tiny를 택했다.
  • 안정성 vs 즉시성/단순함 — 라이다를 별도 프로세스로 격리해 안정성을 얻은 대신, 파일을 거치는 데이터 공유로 인한 약간의 지연(최대 0.1초)과 감시자 구조의 관리 복잡도를 감수했다. "차가 안전장치 없이 갑자기 멈추는 것"보다 "라이다 반응이 0.1초 늦는 것"이 훨씬 나은 선택이었다.
이번 프로젝트의 가장 큰 한계는 한마디로 — 차가 자기 위치를 정확히 모른다는 것.

GPS도, 바퀴 회전 기반 위치 계산도 없다 보니, 장애물 피한 뒤 복귀하기나 흔들림 없이 계속 추적하기 같은 기능들이 전부 "지금 카메라에 뭐가 보이는지"에만 의존하는 대략적인 방식에 머물렀다. "직전 위치 기억하기" 방식은 크기·모양 같은 별도 검증 기준 없이는 한 번 엉뚱한 것에 붙잡히면 계속 그 근처에서만 맴도는 약점이 있는데, 이는 라인트레이싱뿐 아니라 사람 추적(AUTO 모드)에도 똑같이 잠재된 위험이다.

04

향후 진행 방향

계속 이어간다면 아래 순서로 진행하는 걸 추천한다 — 근본 한계(위치 인식)를 건드리는 것이 가장 우선이고, 나머지는 각자 독립적으로 진행 가능하다.

최우선
IMU로 대략적인 위치·방향 추정 추가

이미 달려있는 자이로/가속도 센서(IMU)를 활용해 대략적인 방향이라도 계산하거나, 바퀴 회전 기반 거리 추정을 더하면 "위치를 모른다"는 근본 한계를 넘는 데 가장 효과적이다. 이게 되면 장애물 회피 후 정밀 복귀, 흔들림 없는 추적 두 가지 미해결 문제의 실마리가 동시에 풀린다.

다음
라인 인식에 형태(종횡비) 필터 추가

트랙 로고 오인식 문제는 "가늘고 긴 모양만 선으로 인정"하는 형태 기반 필터로 재시도해볼 것. 크기 비교만으로는 한계가 명확히 확인된 상태라, 모양 정보를 새로 추가하는 방향이 필요하다.

운영
daemon(systemctl) 등록

Jetson이 세션 중 몇 차례 예고 없이 재부팅됐다. 전원이 켜지면 launcher가 자동으로 다시 시작되도록 등록해두면, 재부팅 시에도 폰만으로 복구할 수 있다.
