# 오토카 자율주행 프로젝트 종합 보고서

Jetson Nano를 탑재한 오토카에 자율주행 기능을 구현한 프로젝트. 메인 기능은 두 가지 — **① 사람 따라가기(YOLO)**, **② 라인트레이싱(OpenCV)** — 이고, 두 기능 모두 라이다 장애물 감지를 공통 안전장치로 쓰며, 폰 웹뷰 하나로 모드를 전환하며 조종할 수 있게 만들었다.

이 문서는 **구현한 내용 → 진행하며 생긴 문제점 → 문제 해결 방법 → 정리 및 결론** 순서로 정리한다.

---

## 1. 구현 내용

### 1-1. 두 프로젝트에 사용한 AI/영상처리 방식 비교

| 구분 | 사람 따라가기 | 라인트레이싱 |
|---|---|---|
| 사용 기술 | YOLOv3-tiny (딥러닝 객체 탐지) + TensorRT | OpenCV 영상처리 (규칙 기반) |
| 하는 일 | 카메라 화면에서 "사람"을 찾아 박스로 표시 | 흰 선의 위치(x좌표)를 계산 |
| 연산 위치 | GPU (TensorRT로 가속) | CPU |
| 판단 방식 | 미리 학습된 모델이 "이게 사람이다"를 판단 (학습 기반) | 밝기/모양 규칙으로 "이게 선이다"를 판단 (규칙 기반) |
| 장점 | 사람처럼 복잡한 대상도 정확히 구분 | 가볍고 빠름, 별도 학습 데이터 불필요 |
| 단점 | 모델이 무겁고 GPU 자원을 많이 씀 | 조명/다른 흰색 무늬에 취약함 |

두 기능 모두 카메라를 하나만 쓰기 때문에 동시에 실행할 수 없고, 폰에서 모드를 전환하면 그 모드에 맞는 인식 방식으로 전환되는 구조로 만들었다.

### 1-2. 사람 따라가기 (YOLO)

**YOLO 객체 탐지 최적화 — FPS 6.0 → 30.1 (5배 개선)**

- Jetson Nano에서 YOLOv3-tiny 모델을 **TensorRT**(NVIDIA의 AI 모델 최적화 도구, 학습된 모델을 Jetson 같은 NVIDIA 하드웨어에 맞게 변환해 추론 속도를 높여줌)로 최적화해서 실시간 탐지에 성공했다.
- **FPS(초당 처리 프레임 수)**가 핵심 성과다. 6 FPS는 영상이 뚝뚝 끊겨서 실시간 대응이 어려운 수준이고, 30 FPS는 사람 눈에 부드럽게 느껴지는 "실시간" 수준이다.

**왜 YOLOv4가 아니라 YOLOv3-tiny를 썼는가 (정확도 vs 속도)**

- YOLOv4는 YOLOv3-tiny보다 정확하지만, 파라미터 수로 비교하면 YOLOv4는 약 6,400만 개, YOLOv3-tiny는 약 880만 개로 약 7배 크다. 모델이 크면 연산량도 그만큼 늘어난다.
- YOLOv4를 Jetson Nano에 올리면 FPS가 한 자릿수 이하로 떨어질 것으로 예상된다. 자율주행차에서 초당 2~3프레임이면 장애물을 발견했을 때 이미 늦은 상황이 된다.
- 그래서 정확도를 다소 포기하더라도 실시간(30 FPS)이 가능한 YOLOv3-tiny를 선택했다. 즉각 반응이 필요한 환경에서는 "정확하지만 느린 모델"보다 "약간 덜 정확해도 실시간으로 도는 모델"이 더 실용적이기 때문이다.

**실기 디버깅**

- 차가 안 움직이던 원인 발견/수정: `setSpeed()` 함수가 "이미 주행 중"일 때만 값이 반영되는 함수였는데, 정지 상태에서 호출하면 무효였던 버그. `forward(speed)`로 직접 전달하도록 수정.
- 인식 떨림으로 정지↔전진이 반복되던 문제: 스무딩(값을 급격히 바꾸지 않고 서서히 반영) + 디바운스(짧은 순간의 오차는 무시)로 해결.

**사람 놓치면 카메라 탐색**

- 사람을 놓치면 카메라 서보를 90°(중앙) → 0° → 180° → 90° 순서로 천천히 왕복하며 찾는다 (최대 20회 반복). 그래도 못 찾으면 AUTO 모드를 자동으로 끈다.
- 인식이 깜빡거릴 때마다 탐색 진행 상황이 중앙으로 확 튀는 걸 막기 위해, "연속 3프레임 이상 확인돼야 진짜로 찾은 것"으로 인정하도록 만들었다.

**라이다 안전 회피 통합**

- 라이다가 근접 장애물(1300mm 이내)을 감지하면, 어떤 모드든 상관없이 최우선으로 회피 동작을 하도록 구현하고 실기로 확인했다.

### 1-3. 라인트레이싱 (OpenCV)

**인식 원리**

카메라 화면에서 흰 선을 찾아, 그 선이 화면 중앙에 오도록 조향하는 방식이다. "주변 트랙보다 상대적으로 밝은가"를 기준으로 선을 판단해서(적응형 임계값), 날씨나 노출이 바뀌어도 안정적으로 동작하도록 만들었다.

**오인식 방지**

- 인식 범위(ROI)를 화면 아래쪽 50~55%로 좁혀서 하늘/건물이 분석 대상에서 아예 빠지도록 했다.
- 트랙 표면의 균열이 노이즈로 걸리는 걸 막기 위해, 일정 크기 이상으로 뭉친 덩어리만 선으로 인정하도록 필터를 뒀다.
- 트랙 페인트가 닳아 점선처럼 끊어진 경우가 많아, 끊어진 조각들을 먼저 이어붙인 뒤 판단하도록 처리했다.
- 선 추적 방식은 "양옆 두 선 사이 중앙 유지"가 아니라, 실제 트랙 구조(중앙에 점선 하나)에 맞춰 **가장 크게 잡힌 흰 선 덩어리 하나를 추적**하는 단순한 방식으로 확정했다.
- LINE 모드로 들어가는 순간 카메라를 좌우 정중앙(pan 90°) + 최대한 아래쪽(tilt -10°)으로 자동 정렬해서 버그가 덜 나도록 했다.

**디버그 화면**

라인이 어떻게 인식되고 있는지 확인하기 위해, 요청(새로고침 버튼)할 때마다 현재 인식 중인 화면(선을 초록색으로 표시)을 받아오는 기능을 조종 화면에 내장했다. **스트리밍 방식을 쓰지 않은 이유**는, 영상을 계속 스트리밍하면 처리 부하가 커져서 반응 속도가 크게 느려지기 때문이다.

**처리 속도 개선**

- 선 인식 계산을 절반 해상도로 줄여서 속도를 확보했다.
- 디버그 사진 생성(사람이 볼 때만 필요)도 매 프레임이 아니라 일정 간격으로만 하도록 줄였다.

**장애물 대응 방식**

라인트레이싱 중 장애물을 만나면 **일단 멈췄다가, 장애물이 사라지면 다시 출발**하는 방식으로 구현했다 (자세한 이유는 3장 참고).

### 1-4. 라이다와 AI(YOLO/OpenCV)의 역할 분담 — 안전 우선순위 정리

라이다(장애물 감지)와 AI 인식(YOLO 사람 추적 / OpenCV 라인 추적)은 같은 층위에서 경쟁하는 게 아니라, **라이다가 항상 최우선**으로 개입하는 구조로 교통정리를 했다.

- 매 프레임마다 제일 먼저 라이다 상태(좌/우 근접 장애물 여부)를 확인한다.
- 근접 장애물이 감지되면, 지금 모드가 AUTO든 LINE이든 상관없이 **그 프레임에서는 YOLO/라인 인식 로직을 아예 실행하지 않고** 회피(또는 정지) 동작만 실행한 뒤 다음 프레임으로 넘어간다.
- 장애물이 없을 때만 비로소 각 모드(YOLO 추적 / 라인 추적)의 로직이 실행된다.
- 딱 하나 예외가 "MANUAL_NOLIDAR" 모드다. 벽에 가까이 붙여서 일부러 조종해야 하는 상황을 위해, 이 모드에서만 라이다 개입을 끌 수 있게 만들었다.

정리하면 **라이다 = 안전을 담당하는 최종 결정권자**, **YOLO/OpenCV = 장애물이 없을 때의 평상시 주행 담당**이라는 위계로 나눈 것이다. 이렇게 안 나누고 두 판단을 동시에 반영하려고 했다면, "AI는 전진하라는데 라이다는 멈추라고 하는" 상황에서 어느 쪽을 따를지 애매해졌을 것이다.

### 1-5. 스레드/프로세스 구조

이 프로그램 안에서 "동시에 여러 일을 처리하는 방식"은 세 층으로 나뉘어 있다.

1. **메인 스레드**: 카메라 읽기, YOLO 인식, 라인트레이싱 인식, 모터 제어(`hardware_engine` 함수)가 전부 여기서 실행된다. YOLO의 GPU 연산(pycuda)은 "처음 초기화된 스레드에서만 동작한다"는 제약이 있어서, 반드시 메인 스레드에서 돌려야 했다.
2. **별도 스레드 1개**: 폰과 통신하는 Flask 웹서버가 여기서 돌아간다. 메인 스레드가 카메라/모터 제어로 바쁜 동안에도 폰의 요청(모드 전환, 조향 명령 등)을 받을 수 있어야 하기 때문에 분리했다.
3. **완전히 별도의 프로세스 1개**: 라이다는 스레드가 아니라 아예 다른 프로그램(프로세스)으로 분리했다. 이유는 3장(문제점)에서 자세히 설명한다 — 스레드로 돌리면 GIL 문제로 프로그램 전체가 죽었기 때문이다.

즉 "가벼운 통신 작업(폰)은 스레드로, 무거운 하드웨어 작업(카메라/YOLO)은 메인 스레드로, 불안정한 하드웨어 작업(라이다)은 아예 별도 프로세스로" 나눈 구조다.

### 1-6. 공통 인프라

- **조도센서 + LED 자동 제어**: 조도센서 값을 보고 주변이 어두워지면 LED를 자동으로 켜고, 밝아지면 끄도록 구현했다. 수동 켜기/끄기도 가능하다.
- **폰 핫스팟 + 웹뷰**: 폰의 핫스팟을 공유기 대신 사용하고, 그 네트워크 안에서 폰 브라우저로 접속해 조종할 수 있는 웹뷰(대시보드)를 구현했다.
- **Tmux로 서버 2개 분리 실행**: Jetson 안에서 프로그램을 tmux 세션 2개로 나눠서 돌렸다.
  - `launcher` 세션 (관리자 역할, 포트 5001): 시동/종료 버튼만 처리하는 가벼운 서버. 항상 켜져 있어야 폰에서 버튼이 작동한다.
  - `autocar` 세션 (실제 작업 역할, 포트 5000): 카메라+YOLO+라인트레이싱+라이다가 통합된 메인 프로그램. 코드 수정 후 이 세션만 껐다 켰다 했다.
  - **tmux를 쓴 이유**: SSH 연결이 끊기면 보통 그 안에서 실행 중이던 프로그램도 같이 죽는데, tmux 세션 안에서 실행해두면 SSH가 끊겨도 Jetson 안에서 프로그램이 계속 돌아간다.
  - **세션을 2개로 나눈 이유**: "전원 스위치"(launcher, 리모컨)와 "본체"(메인 프로그램, TV)를 분리하기 위해서다. 둘을 하나로 합쳤다면 메인 프로그램을 재시작할 때 관리자 기능까지 같이 죽어서 "재시작 버튼을 누를 서버 자체가 없어지는" 문제가 생긴다. 분리해두면 메인 프로그램은 몇 번을 껐다 켜도 관리자는 항상 살아있어서 폰으로 계속 제어할 수 있다.

## 2. 진행하며 생긴 문제점

### 2-1. 라이다를 스레드로 돌리니 프로그램 전체가 죽음

처음엔 라이다를 스레드(같은 프로그램 안에서 동시에 여러 일을 처리하는 방식)로 돌렸는데, 카메라/YOLO와 같이 돌아가다가 프로그램 전체가 죽는 문제가 있었다.

### 2-2. 카메라와 라이다가 자원을 동시에 쓰면 멈춤

라이다가 연결되는 순간 카메라/GPU가 이미 자원(전력, 처리 능력)을 쓰고 있으면, 라이다 연결 과정에서 멈추는 경우가 있었다.

### 2-3. 별도 프로세스로 분리해놔도 그 프로세스 자체가 멈춤

라이다를 완전히 별도 프로세스로 분리한 뒤에도, 그 프로세스가 하드웨어 연결(`connect()`) 도중 아무 반응 없이 완전히 멈춰버리는 일이 실제로 있었다.

### 2-4. 라인트레이싱 - 흰 선을 못 찾음

날씨/노출에 따라 흰색의 밝기가 계속 달라져서, "밝기가 특정 값 이상이면 흰색"이라는 절대 기준으로는 흐린 날 흰 선도 인식이 안 되는 문제가 있었다.

### 2-5. 라인트레이싱 - 하늘/건물/트랙 균열을 선으로 착각

인식 범위가 너무 넓어서 하늘/건물 윤곽선까지 분석 대상에 들어갔고, 트랙 표면의 작은 균열도 밝기 차이 때문에 노이즈로 걸렸다.

### 2-6. 라인트레이싱 - 점선이 끊어져서 조각이 너무 작음

트랙 페인트가 닳아 점선처럼 끊어져 있어서, 조각 하나하나가 너무 작아 선으로 인정받지 못하는 경우가 있었다.

### 2-7. 카메라 프레임이 밀리는 문제

처리 속도가 카메라가 찍는 속도를 못 따라가면, 화면에 실시간이 아니라 몇 초 전의 오래된 모습이 계속 나오는 문제가 있었다.

### 2-8. 라인트레이싱 - 트랙 위 로고 같은 무늬를 선으로 착각

트랙에 그려진 원형 로고처럼, 선이 아닌 다른 흰색 무늬가 실제 선보다 크게 잡히면 그쪽을 선으로 잘못 인식하는 문제가 있었다.

### 2-9. 라인트레이싱 - 장애물 피한 뒤 정확히 원래 선으로 복귀하기 어려움

장애물을 피해서 돌아간 뒤 원래 가던 선으로 정확히 복귀하는 기능을 만들고 싶었지만, 이 차에는 GPS나 바퀴 회전 기반 위치 추적(오도메트리)이 없어서 "지금 정확히 어디에 있는지" 알 방법이 없었다.

### 2-10. Jetson이 세션 도중 여러 번 재부팅됨

작업 중 Jetson의 호스트네임이 여러 번 바뀐 것이 확인됐는데(예: `halifax` → `conflagrations` → `monopolizes`), 이는 예고 없이 재부팅됐다는 뜻이다. 재부팅되면 tmux 세션이 전부 사라져서 프로그램이 통째로 꺼진다.

## 3. 문제 해결 방법

### 3-1. (2-1 관련) 스레드 문제 → 별도 프로세스로 분리

원인은 파이썬의 **GIL**이라는 특성 때문이다. 파이썬은 스레드를 여러 개 만들어도 실제로 코드를 실행하는 건 한 번에 딱 하나의 스레드만 할 수 있다. 마치 회의실에 마이크가 하나뿐이라 한 사람만 말할 수 있고, 나머지는 마이크(GIL)를 넘겨받을 때까지 기다려야 하는 것과 비슷하다.

라이다는 하드웨어와 직접 통신하는 C언어 기반 코드(네이티브 라이브러리)를 쓰는데, 이 코드가 하드웨어 응답을 기다리다가 마이크(GIL)를 안 넘겨주고 계속 붙잡고 있는 경우가 있었다. 그러면 카메라/YOLO가 돌아가는 메인 스레드는 마이크를 못 받아 아무것도 못 하게 되어, 프로그램 전체가 멈춘 것처럼 보였다. 게다가 YOLO(GPU 연산)는 "처음 시작한 스레드에서만 작동"하는 제약까지 있어서 더 꼬이기 쉬웠다.

**해결**: 라이다를 스레드가 아니라 **완전히 별도의 프로그램(프로세스)**으로 분리했다. 프로세스는 각자 자기만의 GIL을 갖기 때문에, 라이다 프로세스가 멈춰도 카메라/YOLO 프로세스는 전혀 영향을 받지 않는다.

**프로세스 분리의 장단점**

| | 내용 |
|---|---|
| 장점 | 격리(라이다가 죽어도 메인 프로그램은 무사함), GIL 안 겹침, 문제 생기면 그 프로세스만 강제 종료 후 재시작 가능 |
| 단점 | 프로세스끼리 직접 데이터 공유가 안 돼서 파일(`/tmp/lidar_state.json`)을 거쳐야 함, 그로 인한 약간의 지연(최대 0.1초), 프로세스를 하나 더 띄우는 만큼 더 무거움(메모리/시작 시간), 살아있는지 감시하고 재시작해주는 로직이 별도로 필요함(3-3 참고) |

### 3-2. (2-2 관련) 동시 자원 사용 문제 → 대기 시간 + 타임아웃

라이다 연결 시도 전에 몇 초 대기하고, 연결 시도 자체에 제한시간을 걸어서 — 시간 안에 안 되면 라이다 없이도 나머지(폰 조종, YOLO)는 정상 작동하도록 안전장치를 추가했다.

### 3-3. (2-3 관련) 프로세스 자체가 멈추는 문제 → 감시자-작업자 이중 구조

프로세스 분리는 "멈춰도 다른 기능은 안 죽는다"는 걸 보장하지만 "애초에 안 멈추게" 만들어주지는 않는다. 그래서 감시자(부모) - 실제 작업자(자식) 구조를 한 겹 더 둬서, 자식이 정해진 시간(10초) 안에 연결에 성공하지 못하면 부모가 강제로 죽이고 새 자식으로 재시도하도록 했다. 내부에서 완전히 멈춰도 바깥에서 살릴 수 있게 만든 것이다.

### 3-4. (2-4 관련) 흰 선을 못 찾던 문제 → 상대 밝기 판단 방식

"주변 트랙보다 상대적으로 밝은가"를 보는 방식(적응형 임계값)으로 교체했다. 날씨나 노출이 바뀌어도 선과 트랙 사이의 상대적인 밝기 차이는 유지되기 때문에 훨씬 안정적이다.

### 3-5. (2-5 관련) 하늘/건물/균열 오인식 → 인식 범위 축소 + 노이즈 필터

인식 범위(ROI)를 화면 아래쪽 50~55%로 좁혀서 하늘/건물이 아예 분석 대상에서 빠지게 했고, 일정 크기 이상으로 뭉친 덩어리만 선으로 인정하도록 필터를 추가했다.

### 3-6. (2-6 관련) 점선 끊김 → 조각 이어붙이기

끊어진 점선 조각들을 먼저 이어붙인 뒤 하나의 선으로 판단하도록 처리했다.

### 3-7. (2-7 관련) 프레임 밀림 → 최신 프레임만 보기 설정

GStreamer(카메라 영상 처리 통로)의 마지막 출구(`appsink`)에 "처리 속도가 못 따라가면 오래된 화면은 버려라"는 옵션이 빠져 있었던 것이 원인이었다. 이 옵션을 추가해서 항상 최신 프레임만 보게 만들었다. 카메라를 여는 코드는 프로그램 전체에서 한 곳뿐이라, 이 수정은 라인트레이싱뿐 아니라 AUTO(YOLO) 모드에도 동일하게 적용된다.

### 3-8. (2-8 관련) 로고 오인식 → 미해결, 시행착오만 확인

"크기는 무시하고 직전 위치와 가장 가까운 것 우선"하는 방식으로 바꿔서 시도해봤지만, 오히려 실제 선을 눈앞에 두고도 못 찾는 더 나쁜 문제가 생겨서 검증됐던 기존 방식(크기 큰 것 우선 + 위치 참고)으로 되돌렸다. **이 문제는 최종적으로 해결하지 못했다.** 다음 개선 방향으로는 "가늘고 긴 모양만 선으로 인정"하는 형태(종횡비) 기반 필터가 필요해 보인다.

### 3-9. (2-9 관련) 정밀 복귀 문제 → 정지 후 재출발로 대체

카메라 화면에서 "직전에 봤던 선의 위치"를 기억해뒀다가 참고하는 방식을 시도했지만, 이는 실제 위치 정보가 아니라 화면 속 좌표일 뿐이라 정확하지 않았다. 정밀한 위치 복귀는 포기하고, **장애물이 보이면 일단 멈추고 사라지면 다시 출발**하는 방식으로 대체했다. 회피 동작(후진/회전) 자체도 하지 않기로 했는데, 어차피 정확한 복귀가 안 되는 상황에서 회피 동작까지 하면 트랙을 더 크게 벗어날 위험만 커지기 때문이다.

### 3-10. (2-10 관련) Jetson 재부팅 → 미해결 (daemon 등록 필요)

재부팅 원인 자체는 진단하지 못했다. 재부팅될 때마다 SSH로 직접 들어가서 tmux 세션과 프로그램을 손으로 다시 켜줬다. **daemon(systemctl) 등록**을 하면 재부팅돼도 자동으로 다시 켜지겠지만, 이번 세션에서는 설정하지 못했다. 다만 지금 당장 문제를 일으키는 건 아니고, 데모/장기 운용 전에 해두면 좋은 항목이다.

## 4. 정리 및 결론

### 4-1. 기능별 요약

| 기능 | 담당 AI/기술 | 장애물 대응 |
|---|---|---|
| 사람 따라가기 | YOLOv3-tiny + TensorRT (딥러닝) | 라이다 감지 시 회피 동작 |
| 라인트레이싱 | OpenCV 영상처리 (규칙 기반) | 라이다 감지 시 정지 후 재출발 |

### 4-2. 이번 프로젝트의 두 가지 큰 트레이드오프

1. **정확도 vs 속도** (YOLO 쪽): 더 정확한 YOLOv4 대신, Jetson Nano에서도 실시간(30 FPS)이 가능한 가벼운 YOLOv3-tiny를 선택했다.
2. **안정성 vs 즉시성/단순함** (라이다 쪽): 라이다를 별도 프로세스로 격리해서 안정성(격리)을 얻는 대신, 데이터 공유가 파일을 거쳐야 해서 생기는 약간의 지연(최대 0.1초)과 관리 복잡도(감시자 구조)를 감수했다. "차가 안전장치 없이 갑자기 멈추는 것"보다 "라이다 반응이 0.1초 늦는 것"이 훨씬 나은 트레이드오프였기 때문이다.

### 4-3. 남은 근본적인 한계

이번 프로젝트의 가장 큰 한계는 한마디로 **"차가 자기 위치를 정확히 모른다"**는 것이다. GPS도, 바퀴 회전 기반 위치 계산도 없다 보니, 장애물 피한 뒤 복귀하기나 흔들림 없이 계속 추적하기 같은 기능들이 전부 "지금 카메라에 뭐가 보이는지"에만 의존하는 대략적인 방식에 머물렀다. 같은 이유로 "직전 위치 기억하기" 방식은 크기/모양 같은 별도 검증 기준 없이는 한 번 엉뚱한 것에 붙잡히면 계속 그 근처에서만 맴도는 약점이 있는데, 이는 라인트레이싱뿐 아니라 사람 추적(AUTO 모드)에도 똑같이 잠재된 위험이다.

### 4-4. 다음 단계 제안

- 이미 달려있는 자이로/가속도 센서(IMU)를 활용해서 대략적인 방향이라도 계산하거나, 바퀴 회전 기반 거리 추정 기능을 추가하면 "위치를 모른다"는 근본 한계를 넘는 데 가장 효과적일 것이다.
- 라인트레이싱의 로고 오인식 문제는 "가늘고 긴 모양만 선으로 인정"하는 형태 기반 필터로 재시도해볼 것.
- daemon(systemctl) 등록으로 재부팅 시 자동 복구 기능 추가.
